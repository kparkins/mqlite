# PR1 — Selective CoW + Phase A/B split + bounded retry → EngineFatal

**Date:** 2026-05-10
**Branch:** perf/p1-flip-hoist-cow
**Hardware:** MacBook Pro M3 Pro (12-core, 36 GB)
**Build:** `cargo build --release --example perf_axis` (matrix run);
`--features perf-counters` for the AC #4 / #6 instrumented runs.
**Sidecar JSON:** `2026-05-10-post-pr1.json`

## TL;DR

PR1 lifted same-NS multi-writer throughput by **1.77x** (PR0 4166 dps →
PR1 7373 dps median-of-11), comfortably above the 1.05x AC floor and
the 1.20x stretch goal. The post-merge ratio
`R = same_ns_partitioned@4 / multi_ns_single@4` improved from 0.3823 to
**0.3964**, confirming the same-NS bottleneck has been narrowed — both
legs scaled (1.64x and 1.58x respectively) and the same-NS leg
slightly outpaced multi-NS.

The flip path's old whole-frame `Arc::make_mut` loop (3.19% / 4093 leaf
samples in the spike) is now essentially gone (**0.016% / 11 samples**)
— selective CoW only touches chains carrying a `Pending(txn_id)`
entry. The Phase A/B split with bounded retry exhibits **zero retries
and zero exhaustions** over a 30 s same-NS workload (`flip_retry_rate
= 0.000000`, `flip_retry_exhausted = 0`).

`lock_exclusive_slow` did **not** drop 30% — see "AC #3 — Honest
Reframe" below. The new dominant bottleneck is exclusive-latch
acquisition for the Phase B swap, which PR2 (running-sum cache + dead
scanner removal) is positioned to attack.

## AC #1 — Selective-CoW assertion test

`src/storage/paged_engine/tests/flip_committed_concurrent_observers.rs`
installs 4 pending chains and 8 background committed chains on a
single leaf frame, runs `flip_pending_to_committed_for`, and asserts
via `Arc::ptr_eq` that:

- pending chains' `Arc<VecDeque<VersionEntry>>` were replaced by the
  Phase B swap (selective CoW path), and
- background chains' `Arc` is unchanged (selective CoW skipped them).

This is the load-bearing PR1 invariant. Test passes locally
(`cargo test --release --tests flip_pending_to_committed_only_touches_pending_chains`).

## AC #3 — `lock_exclusive_slow` (HONEST REFRAME)

| metric | spike (PR0.5) | PR1 | delta |
|---|---|---|---|
| % of leaf samples | 3.35% | 7.23% | +3.88 pp |
| samples / 12 s | 358.8 | 419.2 | **+16.9%** |
| samples / insert | 0.104 | 0.141 | **+35.6%** |

**Did NOT meet the 30% drop target.** The reframed reading:

PR1 collapsed the per-insert critical-section work (Arc::make_mut
went 4093 → 11 samples; the whole-frame scan in
`flip_pending_to_committed_for` is gone). With each insert's exclusive
hold time now dramatically shorter, writers complete more inserts per
second (1.77x throughput) and therefore **contend for the exclusive
latch more often per second**. Wall-clock time spent waiting on the
exclusive latch went UP because the workload's *attempt rate* went up.

Per-insert and per-second cost both rose. This is a classic Amdahl
shift: PR1 removed the dominant CPU bottleneck and exposed the next
one — exclusive-latch acquisition for the Phase B swap.

PR2 (running-sum cache + delete legacy whole-frame scanner) is
positioned to attack this directly by avoiding the post-flip
recomputation that currently happens under exclusive latch.

**Falsification clause for PR2.** The Amdahl-shift theory predicts
that PR2's per-write Phase-B exclusive latch hold time should drop
*below* PR1's per-insert baseline (currently 0.141 lock_exclusive_slow
samples per insert). PR2's structural AC
(`phase_b_mean_hold_ns() <= 0.70 × phase_b_mean_hold_ns_at_PR1`)
provides the direct measurement; the per-insert sample-count proxy
above should also drop. If PR2 lands and per-insert
`lock_exclusive_slow` stays flat or rises, the Amdahl theory is wrong
and the contention source must be re-investigated before any further
PRs in this thread.

## AC #4 — Bounded-retry telemetry

30 s `same_ns_single@4` run with `--features perf-counters`:

```json
{"axis":"same_ns_single","writers":4,"docs":150820,"elapsed_secs":30.002429,"docs_per_second":5026.93}
{"perf_counters":{"flip_retry_rate":0.000000,"flip_retry_exhausted":0,
                  "shared_latch_wait_p50_ns":42,"shared_latch_wait_p99_ns":706047}}
```

- `flip_retry_rate` = 0 (target < 0.01) — no Phase B `try_swap_chains_if_unchanged`
  ever observed a `Conflict` outcome in 30 s of 4-writer load.
- `flip_retry_exhausted` = 0 (target == 0, hard EngineFatal gate) —
  no transaction ever reached `MAX_FLIP_RETRIES = 3`. The
  `EngineFatal { PostDurablePendingFlipFailure }` path stays unused
  in normal load.

Throughput here (5026 dps) is below the matrix median (7373 dps)
because the perf-counters histogram instrumentation in
`pin_then_latch`'s shared arm adds a per-shared-pin mutex write to
`hdrhistogram::Histogram<u64>`. **AC throughput is measured WITHOUT
perf-counters** — see AC #5.

The three `*@8` same-NS rows were captured separately and appended to

the lift generalizes to higher writer counts on every same-NS axis
(1.51x–1.74x).

Same-NS lift dominates as expected (the per-NS bottleneck was the
target). Cross-NS axes also benefit (writer commit path is shared).

**Broad-spectrum generalization.** Every multi-NS axis improved
(1.18x–1.36x), not just the targeted same-NS axes. This is a positive
signal that the fix is structural rather than same-NS-specific:
selective CoW + Phase A/B split helps any workload that goes through
`flip_pending_to_committed_for`, regardless of how the writers are
sharded across namespaces. Cross-NS @8 lift is smaller than @4
because the commit path's residual exclusive-latch contention bites
harder at higher concurrency (consistent with the AC #3 reframe
above).

read_find_one regressed by 3.4% which is within prior noise envelopes
(PR0 envelope was 3.76%). Read-only path doesn't go through the flip
machinery; the regression source is unidentified but small.

Envelopes for `same_ns_partitioned@4` (9.25%) and `same_ns_batch@4`
(11.55%) exceeded the 5% noise cap. Median-of-11 still gives a stable
central tendency.

- Reader sustained 156 223 ops/s while sharing the namespace with 4

## Methodology notes

- All matrix runs used the same `perf_axis` lifecycle: process
  wall-time IS the workload window because `examples/perf_axis::main`
  ends with `std::process::exit(0)` to bypass `Client::drop`'s
  per-invocation checkpoint. **DO NOT** remove this bypass without
  re-baselining all downstream PRs — it is required for the
  apples-to-apples compounding-delta math (PR0/PR0.5/PR1/PR2 all
  share the identical lifecycle).
- AC #4 / #6 use `--features perf-counters` which adds a per-shared-pin
  histogram write to a global mutex-guarded `hdrhistogram::Histogram<u64>`.
  This depresses throughput by ~32% under same-NS load. It is
  **opt-in only** and never enabled in the matrix runs that feed
  AC #5 / AC #7.
- The matrix initially captured 8 rows; the three `*@8` same-NS rows
  were added in a follow-up sidecar run and appended into the JSON.
  `read_find_one_under_writers` is captured separately above (it is
  not part of the throughput matrix because it produces both reader
  and writer numbers).
