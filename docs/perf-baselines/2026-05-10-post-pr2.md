# PR2 — Running-sum cache + delete legacy whole-frame scanner

**Date:** 2026-05-10
**Branch:** perf/p2-running-sum-cache
**Hardware:** MacBook Pro M3 Pro (12-core, 36 GB)
**Build:** `cargo build --release --example perf_axis` (matrix run);
`--features perf-counters` for the AC #2 / #3 instrumented runs.
**Sidecar JSON:** `2026-05-10-post-pr2.json`

## TL;DR

PR2 replaces the O(N) whole-frame scanner in
`live_delta_payload_exceeds_leaf_budget` with an O(1) read of a
running-sum cache (`Frame.live_delta_payload_bytes: AtomicU64`)
maintained by every chain mutator. The Amdahl-shift theory from the
PR1 sidecar's AC #3 reframe is **CONFIRMED**: the scanner WAS the
bottleneck — its mean call time drops from **693 ns to 23 ns**
(96.7% reduction).

The per-write install critical section
(`install_phase_b_mean_hold_ns`) drops from **1016 ns to 383 ns**
(62.3% reduction), comfortably under the 30% AC threshold.

Throughput on `same_ns_single@4` median-of-11: PR0 4166.49 -> PR1
7373.30 -> PR2 **7793.48 dps** (full matrix). PR2/PR1 = **1.057x**
(above the 1.05x floor). PR2/PR0 = **1.871x compounded** — nearly
2x raw lift across PR1 + PR2. Stretch (1.25x post-PR1) not hit;
the per-write critical section is now small enough that further
wins require attacking other parts of the per-insert path (B-tree
pathing, latch acquire/release, journal append).

Multi-NS axes all within 5% non-regression bound (most improved).
Post-merge R = 0.3886; the per-commit ceiling has NOT been broken
(target was R > 0.5) — that's the next planning unit.

The legacy whole-frame scanner is **deleted** in this PR per
Principle 3 (single canonical path).

## Note on "Phase B" terminology

PR1 introduced "Phase A/B" for the FLIP path (Phase A = shared-latch
prep + selective Arc::make_mut; Phase B = exclusive-latch
verify-and-install in `try_swap_chains_if_unchanged`). PR1's
sidecar quoted `phase_b_mean_hold_ns: 624` for that FLIP-path
counter — preserved historically but **not** the PR2 baseline.

PR2's `install_phase_b_mean_hold_ns: 1016 -> 383` measures a
**different** code path: the per-write install critical section in
`index_maint.rs:565-617` (the SmoLatchSet-held exclusive page
latch over `with_chain` + `live_delta_payload_exceeds_leaf_budget`).
PR2 does not change FLIP Phase B at all — it attacks the per-write
install Phase B, which is where the scanner lives. The `install_`
prefix on the new counter disambiguates from the FLIP-path
terminology.

## Amdahl-shift confirmation (causal chain)

PR1's AC #3 reframe argued: "lock_exclusive_slow per-insert went
+36% post-PR1 because writers complete 1.77x faster and contend
for the exclusive Phase B latch more often per second; the
per-write critical section is now CPU-bound on the
`live_delta_payload_exceeds_leaf_budget` whole-frame walk."

PR2 attacked exactly that scanner. The numbers form a clean causal
chain:

```
install_phase_b_mean_hold_ns:  1016 -> 383 ns   (-633 ns drop)
live_delta_check_mean_hold_ns:  693 ->  23 ns   (-670 ns drop)
                                              -----
                                               -670 ns of -633 ns
```

The micro-metric drop **fully accounts for** the macro-metric drop
(within instrumentation noise). The scanner *was* the binding cost
in PR1's per-write install critical section; nothing else
contributes meaningfully to the install-section reduction. The
Amdahl-shift theory is confirmed empirically.

## Architectural change

**Before PR2** (per-write install critical section pseudocode):

```rust
// Under exclusive page latch:
page.with_chain(&key, |slot| { *slot = Some(new_chain); })?;
// O(N) walk over every chain on the frame:
let exceeds = page.live_delta_payload_exceeds_leaf_budget()?;
```

The scanner walked `frame.deltas`, computed `chain_live_head_bytes`
inline for each chain, and short-circuited on overflow. Every CRUD
write paid this O(N) cost, where N is the number of resident chains
on the leaf page.

**After PR2:**

```rust
// chains.rs (the SINGLE SOURCE OF TRUTH for the byte formula):
pub(crate) fn chain_live_head_bytes(key: &[u8], chain: &VecDeque<VersionEntry>) -> u64 { ... }

// partition.rs Frame:
pub(super) live_delta_payload_bytes: AtomicU64,

// Mutators incrementally maintain the cache:
//   - with_chain:               delta = bytes(after) - bytes(before)
//   - with_all_chains:          full recompute via frame_live_delta_payload_bytes
//   - replace_leaf_and_chains:  full recompute (Finding A)
//   - reconcile_frame_at:       running sum during retain (Finding B)
//   - try_swap_chains_if_unchanged (Phase B): NO-OP — Pending->Committed
//     preserves footprint; debug_assert_eq! witnesses the invariant.

// Reader collapses to O(1):
pub(crate) fn live_delta_payload_exceeds_leaf_budget(&self) -> Result<bool> {
    let cached = frame.live_delta_payload_bytes.load(Ordering::Acquire);
    Ok(LEAF_HEADER_SIZE as u64 + cached > PAGE_SIZE_LEAF as u64)
}
```

The legacy O(N) scanner body is **deleted**. Single canonical path
per Principle 3.

## AC #3 — `live_delta_check_mean_hold_ns` (Amdahl falsifier)

| | live_delta_check_mean_hold_ns |
|---|---|
| PR1 head (O(N) scanner walk) | **693 ns** |
| PR2 (O(1) Acquire load) | **23 ns** |
| Drop | **96.7%** |
| AC target | <= 50 ns |

This is the surgical micro-metric. Before PR2 the scanner walked
every chain on the page; after PR2 it is a single Acquire-load on
`Frame.live_delta_payload_bytes` plus one comparison. 23 ns matches
the order-of-magnitude expected for an AArch64 atomic load + integer
compare. **The Amdahl-shift theory from the PR1 AC #3 reframe is
confirmed**: the scanner *was* the per-insert bottleneck, and PR1's
+36% per-insert increase in `lock_exclusive_slow` was correctly
diagnosed as a queueing artifact of faster writes contending more
often, not as a real per-write hold-time regression.

## Cache invariant test (THE primary correctness verdict)

`src/storage/buffer_pool/tests/running_sum_cache_invariant.rs`
contains three tests:

1. **`stress_10k_mutations_keep_cache_consistent`**: seeds a single
   page with random chains, then runs 10 000 seeded random
   mutations through `with_chain_under_latch`,
   `with_all_chains_under_latch`, and (every 250 iterations)
   `replace_leaf_and_chains`. After every mutation, the cached
   value is compared against a fresh recompute via
   `frame_live_delta_payload_bytes(&frame.deltas)`. Divergence
   panics with the action + key + before/after cached + fresh.
   **Result: 10000 iterations, zero divergence.**

2. **`phase_b_swap_preserves_cache`**: installs N chains with
   `Pending(txn_id)` heads, snapshots the cache, runs
   `try_swap_chains_if_unchanged` to flip them all to
   `Committed`, asserts cache value is unchanged. Witnesses the
   `try_swap_chains_if_unchanged` debug_assert invariant.
   **Result: pass.**

3. **`replace_leaf_and_chains_recomputes_cache`**: exercises audit
   Finding A (the checkpoint dirty-leaf reconcile path that
   bypasses `with_all_chains`). Replaces both the leaf bytes and
   the chain map atomically; asserts the cache equals
   `frame_live_delta_payload_bytes(&new_chains)` post-replace.
   **Result: pass.**

Without these three tests, the cache could diverge from
`frame.deltas` undetected and silently corrupt the
overflow-decision in `live_delta_payload_exceeds_leaf_budget`. The
10 000-iteration stress test is the load-bearing correctness proof.

## Audit findings (frame-load path bypasses)

PR2 design started with an audit of every `frame.deltas` mutation
site in non-test src/ to identify bypasses of the latched mutator
unification (PR0.5). Two findings:

**Finding A — `BufferPool::replace_leaf_and_chains` (mod.rs)** — the
checkpoint dirty-leaf reconcile path holds an exclusive page latch
then directly assigns `frame.deltas = retained_chains;`, bypassing
`with_all_chains`. PR2 wires a fresh recompute into this path. The
`replace_leaf_and_chains_recomputes_cache` test exercises it.

**Finding B — `Partition::reconcile_frame_at` (partition.rs)** — the
eviction-prep prune walks `frame.deltas.retain(|key, chain_arc| {
... Arc::make_mut ... })` directly. Currently safe-by-call-sequence
(only ever called from `pin_page_reconciling` immediately before
`evict_frame`), but the contract is fragile. PR2 folds a running-sum
recompute into the existing retain pass — the cache invariant is now
self-correcting at this site without depending on the call sequence.
~5 LOC, cold path.

## Memory ordering

- `Frame.live_delta_payload_bytes: AtomicU64`
  - Writers: `load(Acquire)` -> arithmetic -> `store(Release)` under
    the exclusive page latch (which already serializes mutator vs
    reader).
  - Readers: `load(Acquire)` under the exclusive page latch.
  - The Acquire/Release pair is **defense-in-depth**: the latch
    already provides the happens-before edge, so Relaxed would be
    sufficient for current callers. But Acquire/Release makes the
    cache safe to read under a shared latch without a correctness
    migration if a future PR chooses to relax the read.
- `INSTALL_HOLD_NS_TOTAL` / `INSTALL_WRITES` /
  `LIVE_DELTA_CHECK_NS_TOTAL` / `LIVE_DELTA_CHECK_CALLS`: pure
  observability counters, `fetch_add(_, Ordering::Relaxed)`.
  Documented in source.

## AC #10 — Post-merge R direction

`R = same_ns_partitioned@4 / multi_ns_single@4` (informational
metric per spike #6 to characterize the per-commit ceiling).

| | same_ns_partitioned@4 | multi_ns_single@4 | R |
|---|---|---|---|
| PR0 | 3844.57 | 10057.52 | 0.3823 |
| PR1 | 6302.63 | 15900.77 | 0.3964 |
| **PR2** | 6226.77 | 16022.78 | **0.3886** |

R is **flat across PR1 and PR2**, hovering near 0.38–0.40. The
spike's threshold for "per-commit ceiling broken" was R > 0.5;
**neither PR1 nor PR2 reached it**. This was expected: PR2 attacks
the per-WRITE critical section (the cache replaces a per-write
scanner). The per-COMMIT ceiling is a separate constraint
(durability-thread interaction, journal append serialization). The
deferred follow-up #5 from spike #6 is the planning unit that
would push R above 0.5.

R direction interpretation:
- 0.3823 -> 0.3964 (PR1): same-NS leg scaled slightly more than
  multi-NS leg, edging R up.
- 0.3964 -> 0.3886 (PR2): both legs scaled by similar ~3-5%
  (1.057x vs 1.008x); R drifted ~0.008 down. Within noise.

The takeaway: PR1 + PR2 lifted the floor of all axes substantially
(1.6-1.9x same-NS, 1.3-1.6x multi-NS) but did NOT change the
ceiling shape. The next planning unit attacks the ceiling.
