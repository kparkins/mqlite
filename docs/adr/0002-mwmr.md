# ADR 0002 — v1 MWMR: namespace-level write lanes + published-snapshot reads

**Status:** Proposed.
**Date:** 2026-04-18.

---

## 1. Context

mqlite's T9 MVCC implementation (ADR 0001) delivers true snapshot isolation
for readers via `ReadView` + per-document version chains. However, both reads
and writes still acquire a single engine-global mutex
(`PagedEngine::inner: Mutex<BpBackend>`). This means:

- A long read blocks all concurrent writes, and vice versa.
- Writers on completely independent namespaces serialize unnecessarily.
- `FullSync` commits fsync the main file via a checkpoint, putting an
  expensive disk operation on every write's hot path.

The MVCC machinery is already in place; this ADR pins the v1 contract for
removing those remaining serialization points.

---

## 2. Decision drivers

1. **Read-path concurrency.** Reads must not acquire any mutex. A snapshot
   load must be a single atomic pointer load (`ArcSwap::load()`), not a
   lock acquisition.

2. **Cross-namespace write concurrency.** Writers on different namespaces
   must overlap in wall-clock time. Writes on the same namespace serialize
   (correct; B-tree page mutation is not safe to interleave within a
   namespace today).

3. **Conservative DDL in v1.** Schema changes (`create_namespace`,
   `drop_namespace`, `drop_index`) take a global metadata write-lock.
   `create_index` is the sole exception (see §4).

4. **Commit ordering.** `commit_ts`, `publish_ts`, and journal-append order
   must be identical across all concurrent commits. Readers must never
   observe a partially-published commit.

5. **Journal-durable `FullSync`.** A `FullSync` commit must fsync the journal
   only. Main-file checkpoint moves off the hot path (admin/background).

---

## 3. Architecture summary

### 3.1 Latest-published-snapshot reads

A field `published: ArcSwap<PublishedSnapshot>` replaces all mutable
read-path state:

```
PublishedSnapshot {
    publish_ts:           HlcTs,
    catalog_root_page:    u32,
    catalog_root_level:   u8,
    namespaces:           HashMap<String, NamespaceSnapshot>,
}

NamespaceSnapshot {
    data_root_page:   u32,
    data_root_level:  u8,
    indexes:          Vec<PublishedIndex>,
}

PublishedIndex {
    name:         String,
    root_page:    u32,
    root_level:   u8,
    key_pattern:  Document,
    unique:       bool,
    sparse:       bool,
    multikey:     bool,
    state:        IndexState,   // Building | Ready
}
```

Every read operation does:

```
let snap = shared.published.load();   // atomic pointer load, no lock
let ns   = snap.namespaces.get(ns)?;
let tree = BTree::open(store, ns.data_root_page, ns.data_root_level);
let view = ReadView::open(registry, snap.publish_ts, txn_id);
btree_collscan(&tree, filter, &view, Some(&probe))
```

**Acceptance gate — snapshot stability**: reads MUST NOT cache a
`PublishedSnapshot` across operations. Each operation calls `load()` once
at its start. Any implementation that holds a `snap` reference across an
`await` point or across a public API boundary fails this gate.

### 3.2 Namespace-level write lanes

```rust
ns_lanes: DashMap<String, Arc<Mutex<()>>>
```

Before entering a write operation on namespace `ns`:

1. Look up (or insert) the lane for `ns` in `ns_lanes`.
2. `lane.lock()` — blocks only writers on the same namespace.

Writers on different namespaces proceed concurrently. Writers on the same
namespace serialize.

### 3.3 Global metadata `RwLock` for DDL

```rust
metadata: RwLock<MetadataState>   // MetadataState holds the live catalog
```

All DDL operations (`create_namespace`, `drop_namespace`, `drop_index`,
`checkpoint`, `backup`) take `metadata.write()` for their full duration.
This is intentionally conservative in v1.

CRUD operations take `metadata.read()` only long enough to resolve the
namespace's lane (or detect namespace absence), then release the read guard
before acquiring the lane.

`create_index` is the sole exception — see §4.

### 3.4 Commit ordering

The commit arm inside a write operation must follow this exact sequence:

```
1.  lane.lock()                               — already held from CRUD entry
2.  stage primary writes + sec-index writes
3.  commit_seq.lock()                         — global commit serialization
4.  commit_ts = oracle.commit()               — HLC timestamp, monotone
5.  install_pending_primary(commit_ts, ...)   — tag versions with commit_ts
6.  handle.flush()                            — non-commit dirty frames
7.  handle.append_chain_commit(commit_ts, …)  — journal frame
8.  handle.commit_txn(...)                    — legacy page-state record
9.  build new PublishedSnapshot from old + this txn's root deltas
10. shared.published.store(Arc::new(snap))    — atomic publish
11. drop commit_seq
12. drop lane
```

`commit_ts`, `publish_ts` (= `snap.publish_ts`), and journal-append order
are all equal. A reader that loads `published` after step 10 sees all
effects of this commit and no effects of commits that have not yet reached
step 10.

### 3.5 Journal-durable `FullSync`

`FullSync` = fsync the journal file only (step 7 above). Main-file
checkpoint is not performed on the commit hot path; it is triggered by
`Client::checkpoint()` (admin) or a background task.

`backup` still performs a full checkpoint before copying.

### 3.6 `insert_many` interleaving

Under namespace lanes each document insertion is its own transaction.
Partial progress from an `insert_many` call is visible to other writers
and readers on the same namespace after each individual commit. If a later
document in the batch fails, earlier inserts are already durable and
visible — they are not rolled back. Batch atomicity for multi-call public
APIs is out of scope for v1.

## 4. `create_index` — Reserve/Build/Commit (three-phase)

`create_index` on namespace A must not block writers on namespace B. The
build scan can be long; holding `metadata.write()` for its duration would
serialize all DDL globally for an unbounded time.

Three-phase protocol:

1. **Reserve** (under `metadata.write()`):
   Allocate the index root page; write an `IndexEntry { state: Building }`
   into the catalog; publish a new `PublishedSnapshot` carrying the
   Building index; release `metadata.write()`.

2. **Build** (under `ns_lanes[ns].lock()`, NOT `metadata.write()`):
   Scan the current data tree and populate the index B-tree. Concurrent
   writers on other namespaces are completely unaffected. Concurrent
   writers on the target namespace serialize on the lane and
   dual-write: any index with `state: Building` on their namespace
   receives new keys alongside each primary write. Queries ignore
   `Building` indexes (fall back to collscan).

3. **Commit** (under `metadata.write()`):
   Flip `state: Building → Ready`; publish a new `PublishedSnapshot`;
   release `metadata.write()`.

## 5. Lock order

Extending ADR 0001's lock order (positions 1–6) with v1-MWMR additions:

```
0.5. commit_seq mutex           — serializes commit_ts allocation (v1-MWMR)
1.   history-store partition mutex
1.5. DeferredFreeQueue::pending
2.   AllocatorHandle::state
2.5. ns_lanes[ns] mutex         — per-namespace write lane (v1-MWMR)
3.   32 KB main partition mutex
4.   4 KB main partition mutex
5.   ReadViewRegistry mutex
6.   PagedEngine::inner Mutex   — retired (replaced by namespace lanes)
```

`metadata: RwLock<MetadataState>` is acquired at the CRUD entry point,
released before any inner lock is taken (no lock is held across the
`metadata.read()` release when acquiring `ns_lanes`). DDL paths that hold
`metadata.write()` do not hold any inner lock simultaneously (except
`commit_seq` during the snapshot publish step).

`ns_lanes` is acquired inside `metadata.read()` — always release the
metadata read guard before the acquire if upgrading to `metadata.write()`.

## 6. Acceptance gates

1. **Read-path mutex-free**: `rg 'inner\.lock' src/storage/paged_engine.rs`
   inside the five read-path functions (`find`, `find_one`, `count`,
   `list_indexes`, `list_namespaces`) returns nothing.

2. **Cross-namespace overlap**: test `mwmr_namespace_lanes` — two threads
   writing to different namespaces overlap in wall-clock time; two threads
   on the same namespace serialize.

3. **Commit ordering**: 1000 mixed-namespace commits verify that
   `commit_ts == publish_ts` and journal-append order matches `commit_ts`
   order for every commit.

4. **Snapshot-stability invariant**: test `reader_stable_across_other_ns_commits`
   remains green. A `ReadView` opened before N concurrent commits on other
   namespaces sees byte-identical results to what it would have seen at `read_ts`.

5. **`insert_many` per-insert durability**: `insert_many` is per-insert durable
   (not all-or-nothing). Earlier inserts in a batch remain visible if a later
   insert fails.

6. **FullSync hot-path benchmark**: `FullSync` insert p50 latency reflects
   journal-only fsync (no checkpoint on the commit hot path).

---

## 7. Out of scope for v1

- Same-namespace multi-writer B-tree concurrency (multiple concurrent
  writers within one namespace).
- Batch atomicity for multi-call public APIs (`insert_many` all-or-nothing).
- Serializable isolation (snapshot isolation is the target).
- Online index build that allows concurrent unrelated writes on the target
  namespace during the build scan (`create_index` requires the namespace
  lane for the build phase; same-namespace writes wait).

---

## 8. Consequences

- `PagedEngine::inner: Mutex<BpBackend>` is retired. Reads and writers on
  different namespaces never block each other.
- `writer_lock: Mutex<()>` in `ClientInner` is retired; busy-timeout /
  busy-handler behavior moves to namespace-lane acquisition.
- `insert_many` is per-insert durable (not all-or-nothing). Earlier inserts
  in a batch remain visible if a later insert fails.
- `create_index` build scan holds the target namespace's lane for its full
  duration. Other-namespace writers are unaffected. Same-namespace writers
  wait for the build to complete (same behavior as any other same-namespace
  write, just longer).
- Cross-process locking is unchanged. The OS advisory file lock remains
  exclusive. MWMR is in-process only.
