# ADR 0001 — WiredTiger-style MVCC for mqlite

**Status:** Implemented.
**Date:** 2026-04-16.

---

## 1. Context

This ADR records the decision to implement WiredTiger-style MVCC in
mqlite, separating the journal (crash recovery) from MVCC (concurrent
read visibility). Full component list lives in `docs/MVCC.md`.

---

## 2. Decision drivers

1. **No reader / writer blocking.** Any reader opens a `ReadView` at the
   timestamp oracle's current value and walks per-document version
   chains in memory. Writers do not block readers; readers do not block
   writers.

2. **True snapshot isolation.** A `ReadView` pins a `read_ts`; every
   observed version satisfies `start_ts <= read_ts < stop_ts`. Readers
   always see a point-in-time consistent view.

3. **Fail-stop on ordering violations.** A Hybrid Logical Clock (HLC)
   oracle issues commit timestamps. On open, the oracle is floored to
   the successor of the last durable commit recorded in the journal,
   so post-recovery commits are strictly above every pre-crash commit.

4. **Memory-bounded version retention.** Reconciliation (on eviction)
   drops versions older than `oldest_required_ts`. A dedicated B-tree
   history store accepts chains that are evicted before
   `oldest_required_ts` has advanced past them; readers fall back to
   the history store on in-memory misses.

5. **Crash recovery unchanged.** The journal still
   records reconciled on-disk page state. MVCC version chains are
   in-memory only and rebuilt lazily from the journal on reopen.

6. **Atomic multi-chain commits.** Primary-key updates, secondary-index
   updates, and refcount deltas for overflow-page pins all land under a
   single `commit_ts`. Readers at `read_ts < commit_ts` see the
   pre-update state across the whole txn; readers at `read_ts >=
   commit_ts` see the post-update state. No half-committed witness
   exists.

7. **Force-expire barriers for non-MVCC-safe ops.** `drop_collection` is
   v1's only blocking barrier — it force-expires every live `ReadView`
   globally, waits for `pin_ops_in_flight == 0`, then invokes
   `free_subtree` under writer serialization.

## 3. Architecture summary

Full component list lives in `docs/MVCC.md`. Top-level:

- `src/mvcc/timestamp.rs` — HLC oracle (`Ts { physical_ms, logical }`),
  monotonic under clock regressions, floored on recovery.
- `src/mvcc/version.rs` — `VersionEntry`, `OverflowRef` (RAII refcount
  handle over overflow-chain first pages), `VersionData::{Inline,
  Overflow}`.
- `src/mvcc/read_view.rs` — `ReadView` (poison + `pin_ops_in_flight`),
  `ReadViewRegistry` (position 5 in the global lock order),
  `ChainSnapshot` (S13 atomic-handoff pin protocol).
- `src/mvcc/transaction.rs` — `WriteTxn::begin` (drains
  `DeferredFreeQueue` before allocating), staged primary/sec-index
  writes, atomic commit at one `commit_ts`, panic-rollback via RAII.
- `src/mvcc/deferred_free.rs` — writer-drained queue of overflow
  first-pages whose refcount hit zero on the reader path.
- `src/storage/buffer_pool.rs` — reconciliation on eviction walks every
  chain against `ReadViewRegistry::oldest_required_ts()`, drops aged
  entries, pushes cold entries into the history store.
- `src/storage/history_store.rs` — kind-tagged keys
  `(ns_id, kind, user_key, start_ts_BE)`; reader probes on chain miss;
  GC pass runs from `checkpoint()`.
- `src/journal/` — crash recovery only. `ChainCommit` frames carry
  `commit_ts + refcount_deltas + page_writes + pending_sec_index`, so
  recovery restores atomicity across primary and sec-index chains
  without ever re-deriving in-memory state.

### Lock order (position 1 = outermost)

```
1.   history-store partition mutex
1.5. DeferredFreeQueue::pending
2.   AllocatorHandle::state
3.   32 KB main partition mutex
4.   4 KB main partition mutex
5.   ReadViewRegistry mutex
6.   writer serialization mutex (PagedEngine::inner)
```

Readers take 1.5 briefly on drop; writers take 5 → 6. `drop_collection`
acquires 6 first, then briefly snapshots 5 for `force_expire_all` —
because the registry mutex is never held across any downstream
acquisition this violates the *total* order but not liveness.

---

## 5. Consequences

- Readers and writers do not block each other. MVCC visibility
  (`ReadView` + version chains) ensures reads see a snapshot-consistent
  view. The engine-level mutex (`PagedEngine::inner`) remains on the
  write path; read-path mutex removal is tracked in [ADR 0002](0002-mwmr.md).
- Memory cost per dirty page rises by the size of its live version
  chains. Bounded by reconciliation under `oldest_required_ts`.
- `drop_collection` has a force-expire barrier — any caller holding
  a long-lived `ReadView` across a drop will see
  `Error::ReadViewExpired` on its next operation and must open a new
  view. Reads on the dropped namespace return zero rows, matching
  MongoDB's drop-is-a-global-barrier semantics.
