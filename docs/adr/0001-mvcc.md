# ADR 0001 — WiredTiger-style MVCC for mqlite

**Status:** Accepted — landed across commits T0…T9 (MVCC branch).
**Date:** 2026-04-16.
**Authors:** MVCC working group (see `.omc/plans/mvcc-wiredtiger.md`).

---

## 1. Context

mqlite shipped a single-writer / shared-readers concurrency model where
every reader took a shared lock on the SQLite-style WAL for the lifetime
of a scan. The WAL was doing two unrelated jobs:

1. **Crash recovery** (correct, should be preserved).
2. **Snapshot isolation** (incorrect; it was never designed for that).

Symptoms in the wild:

- A long `find()` blocked every concurrent write.
- Heavy write workloads caused unbounded read latency and vice versa.
- Only the latest committed version of each document existed — no room
  for multi-statement snapshot isolation.

MongoDB's WiredTiger solves the same problem with per-document in-memory
version chains. This ADR records the decision to port that design into
mqlite and to cleanly separate the journal (crash recovery) from MVCC
(concurrent read visibility).

---

## 2. Decision drivers

1. **No reader / writer blocking.** Any reader opens a `ReadView` at the
   timestamp oracle's current value and walks per-document version
   chains in memory. Writers do not block readers; readers do not block
   writers.

2. **True snapshot isolation.** A `ReadView` pins a `read_ts`; every
   observed version satisfies `start_ts <= read_ts < stop_ts`. Readers
   always see a point-in-time consistent view.

3. **Fail-stop on ordering violations.** A Hybrid Logical Clock (HLC)
   oracle issues commit timestamps. On open, the oracle is floored to
   the successor of the last durable commit recorded in the journal,
   so post-recovery commits are strictly above every pre-crash commit.

4. **Memory-bounded version retention.** Reconciliation (on eviction)
   drops versions older than `oldest_required_ts`. A dedicated B-tree
   history store accepts chains that are evicted before
   `oldest_required_ts` has advanced past them; readers fall back to
   the history store on in-memory misses.

5. **Crash recovery unchanged.** The journal (formerly "WAL") still
   records reconciled on-disk page state. MVCC version chains are
   in-memory only and rebuilt lazily from the journal on reopen.

6. **Atomic multi-chain commits.** Primary-key updates, secondary-index
   updates, and refcount deltas for overflow-page pins all land under a
   single `commit_ts`. Readers at `read_ts < commit_ts` see the
   pre-update state across the whole txn; readers at `read_ts >=
   commit_ts` see the post-update state. No half-committed witness
   exists.

7. **Force-expire barriers for non-MVCC-safe ops.** `drop_collection` is
   v1's only blocking barrier — it force-expires every live `ReadView`
   globally, waits for `pin_ops_in_flight == 0`, then invokes
   `free_subtree` under writer serialization.

## 3. Alternatives considered

| Option | Verdict | Why not |
|--------|---------|---------|
| **Keep RwLock + latest-only page cache.** | ✗ | Status quo. Long readers starve writers. No snapshot isolation. |
| **Per-document lock + MVO (multi-version on disk).** | ✗ | Write amplification unacceptable for our page-per-txn model. |
| **Postgres-style heap + visibility map.** | ✗ | Would require rewriting the allocator. Heap bloat is worse than version-chain drift for small-doc workloads. |
| **WiredTiger-style in-memory chain + history B-tree.** | ✓ | Chosen. Matches MongoDB exactly, minimises write amplification, recovers lazily, and degrades gracefully (chain → history store → disk). |

## 4. Architecture summary

Full component list lives in `docs/MVCC.md`. Top-level:

- `src/mvcc/timestamp.rs` — HLC oracle (`Ts { physical_ms, logical }`),
  monotonic under clock regressions, floored on recovery.
- `src/mvcc/version.rs` — `VersionEntry`, `OverflowRef` (RAII refcount
  handle over overflow-chain first pages), `VersionData::{Inline,
  Overflow}`.
- `src/mvcc/read_view.rs` — `ReadView` (poison + `pin_ops_in_flight`),
  `ReadViewRegistry` (position 5 in the global lock order),
  `ChainSnapshot` (S13 atomic-handoff pin protocol).
- `src/mvcc/transaction.rs` — `WriteTxn::begin` (drains
  `DeferredFreeQueue` before allocating), staged primary/sec-index
  writes, atomic commit at one `commit_ts`, panic-rollback via RAII.
- `src/mvcc/deferred_free.rs` — writer-drained queue of overflow
  first-pages whose refcount hit zero on the reader path.
- `src/storage/buffer_pool.rs` — reconciliation on eviction walks every
  chain against `ReadViewRegistry::oldest_required_ts()`, drops aged
  entries, pushes cold entries into the history store.
- `src/storage/history_store.rs` — kind-tagged keys
  `(ns_id, kind, user_key, start_ts_BE)`; reader probes on chain miss;
  GC pass runs from `checkpoint()`.
- `src/journal/` — crash recovery only. `ChainCommit` frames carry
  `commit_ts + refcount_deltas + page_writes + pending_sec_index`, so
  recovery restores atomicity across primary and sec-index chains
  without ever re-deriving in-memory state.

### Lock order (position 1 = outermost)

```
1.   history-store partition mutex
1.5. DeferredFreeQueue::pending
2.   AllocatorHandle::state
3.   32 KB main partition mutex
4.   4 KB main partition mutex
5.   ReadViewRegistry mutex
6.   writer serialization mutex (PagedEngine::inner)
```

Readers take 1.5 briefly on drop; writers take 5 → 6. `drop_collection`
acquires 6 first, then briefly snapshots 5 for `force_expire_all` —
because the registry mutex is never held across any downstream
acquisition this violates the *total* order but not liveness.

## 5. Acceptance bullets shipped

Unit + integration coverage by task:

- T2 — HLC monotonicity, clock regression, logical overflow,
  endianness round-trip.
- T3 — visibility truth-table, registry open/drop/min, CAS-loop incref
  saturation, deferred-free drain.
- T3.5 — split migration, `free_leaf` merge-path guard at
  `src/storage/btree.rs:1281, 1308`.
- T3.75 — `ChainSnapshot::new` poison pre-check and post-check.
- T4 — `ReadViewRegistry` deadlock gate, registry stress (1000
  concurrent open/drop).
- T5' — remove RwLock read path, `WriteTxn::begin` drains deferred
  free queue, atomic commit across primary + sec-index + refcount.
- T6 — reconcile drop-count + RAII decref + deferred-free drain.
- T7 — history store round-trip, HLC oracle floor on reopen,
  kind-tagged keys.
- T8 — GC + 12 mandatory counters + 5 diagnostic counters +
  force-expiry.
- **T9** — this commit. E2E tests, `drop_collection` barrier protocol,
  feature-flag removal, ADR.

`drop_collection` barrier semantics are covered by
`tests/mvcc_e2e.rs::drop_collection_barrier`:

- Seed a collection, register N=5 `ReadView`s.
- Invoke `db.drop_collection("victims")`.
- Assert every pre-opened view is `poisoned` and that
  `view.check_active()` returns `Error::ReadViewExpired`.
- Assert reads on the dropped collection return zero rows and the
  namespace is gone from the catalog.
- Assert no deadlock (test completes under the 60-second harness
  budget).

## 6. Benchmarks and long-running gates

Four acceptance gates from the T9 plan require external infrastructure
that is not present in this crate. They ship as `#[ignore]`'d tests in
`tests/mvcc_e2e.rs` and are explicitly deferred to post-v1:

1. **YCSB-A 3× throughput / 5× p99 gate** — requires a pre-MVCC master
   baseline binary for the comparison. The MVCC-side run is provided;
   an external script would diff it against a separate master run.
2. **Tombstone-elision ≥ 95% gate** — mqlite ticks
   `tombstone_hits_skipped_total` but does not currently expose a
   discrete `tombstone_index_entries_generated` counter, so the full
   ratio is not directly assertable. The weaker assertion that *some*
   tombstone hits are skipped under churn is enforced.
3. **Real `kill -9` crash test** — requires a child-process harness.
   The shipped test uses `drop(Client)` without `close()` as a
   simulation; the commit-count-preservation assertion still holds.
4. **24 h soak** — the shipped proxy runs for 10 seconds with writers,
   readers, and ReadView churn; the `overflow.pages_in_use`-drift check
   is enforced.

Also deferred to post-v1 (plan §T8 residuals):

- `mvcc.version_chain_depth_p99` production sampler in `checkpoint` /
  `reconcile_frame_at`.
- `mvcc.history_store_bytes` production sampler summing history-store
  B-tree bytes.
- `mvcc.reconcile.duration_ms_p99` tick site wrapping
  `reconcile_frame_at`.
- Force-expiry auto-trigger when history store exceeds a size hard-cap
  (current default: `HISTORY_STORE_HARD_CAP_BYTES = 512 MiB`).

All four are plumbed via existing counter APIs — the sampling sites
are the only missing piece. Follow-up tickets will thread them in.

## 7. Consequences

- No more WAL-as-concurrency. Readers and writers currently serialize
  on `PagedEngine::inner: Mutex<BpBackend>`, but MVCC visibility
  (`ReadView` + version chains) ensures reads see a snapshot-consistent
  view. Moving that mutex off the read path is planned for v1-MWMR
  (see [ADR 0002](0002-mwmr.md)).
- On-disk file format is unchanged. A post-T9 mqlite reads a pre-T0
  file transparently (the `-wal` sidecar rename is the only
  migration-time change, handled automatically on open).
- Memory cost per dirty page rises by the size of its live version
  chains. Bounded by reconciliation under `oldest_required_ts`.
- `drop_collection` now has a force-expire barrier — any caller holding
  a long-lived `ReadView` across a drop will see
  `Error::ReadViewExpired` on its next operation and must open a new
  view. Reads on the dropped namespace return zero rows, matching
  MongoDB's drop-is-a-global-barrier semantics.
